## 2. 初始 Review 结论

初始 review 的 7 个 finding 都已编号归档到 [`docs/archive/V5/review_resolution.md`](/D:/构想/stock-assistant/docs/archive/V5/review_resolution.md)：

- `V5-001`：手动创建计划无法正常选股
- `V5-002`：`PlanAgent` 只做结构校验，没有复用业务校验
- `V5-003`：`generate` 错误语义混乱，用户侧问题被映射成 `500`
- `V5-004`：前端缺少已保存 `pending` 计划的编辑入口
- `V5-005`：页面错误提示不完整，存在“点击后没反应”
- `V5-006`：`total_capital` / `status` 默认契约分叉
- `V5-007`：测试偏 service 层，API 语义和前端路径缺口明显

## 3. 根因分析

### 3.1 后端

- 计划保存校验和 LLM proposal 校验分散在不同位置，没有单一契约层。
- `PlanAgent` 只关心 JSON 能否解析，没有校验 proposal 是否满足业务规则。
- `user_config` 的默认值只存在于规格和前端兜底里，没有真正落到运行期行为和迁移层。
- API 没有把 `validation / not_found / upstream` 稳定映射到 `400 / 404 / 502`。

### 3.2 前端

- `PlanView` 和 `PlanGenerateView` 各自维护一套草稿编辑逻辑，导致手动创建、generate 编辑和已保存计划编辑逐步分叉。
- `searchStocks()` 的响应契约没有统一在 API 层解包，直接把旧返回结构泄漏给页面。
- 页面依赖 store 持有错误状态，但没有强制在 UI 上展示。
- 时间显示直接基于字符串处理，没有统一格式化。

### 3.3 交付与复核

- 缺少“review finding -> 修复动作 -> 变更文件 -> 验证结果”的成套归档。
- 复核时需要同时翻 diff、测试、文档，缺少单一入口。

## 4. 本轮重构方案

### 4.1 后端契约收口

- 新增 [`backend/app/services/plan_contract.py`](/D:/构想/stock-assistant/backend/app/services/plan_contract.py)，统一处理：
  - 计划 payload 校验
  - proposal 校验
  - 统一错误对象
  - 手动 fallback 响应
  - DB -> API 的序列化
- `PlanAgent` 改成：
  - 先提取第一个合法 JSON 块
  - 再解析 proposal 列表
  - 最后复用同一套业务规则校验 proposal
- `generate` 返回判别式结构：
  - `status: "ok"`
  - `status: "manual_fallback"`
- API 层稳定映射：
  - `POST /api/plan` -> `201`
  - `PUT/PATCH` 不存在资源 -> `404`
  - 参数非法 -> `400`
  - 上游异常 -> `502`
- `total_capital` 默认值视为 `"0"`，由 migration + 运行期 `ensure_default_configs()` 双保险兜住。

### 4.2 前端编辑流统一

- 新增共享编辑器 [`frontend/src/components/plan/PlanEditor.vue`](/D:/构想/stock-assistant/frontend/src/components/plan/PlanEditor.vue)。
- 新增 [`frontend/src/utils/plan.ts`](/D:/构想/stock-assistant/frontend/src/utils/plan.ts) 统一：
  - draft 构造
  - payload 生成
  - 止盈档位 clone
  - 本地时间格式化
- [`frontend/src/views/PlanView.vue`](/D:/构想/stock-assistant/frontend/src/views/PlanView.vue) 收口为：
  - 列表
  - 手动创建
  - `pending` 计划编辑
  - 状态流转
  - 详情抽屉
- [`frontend/src/views/PlanGenerateView.vue`](/D:/构想/stock-assistant/frontend/src/views/PlanGenerateView.vue) 收口为：
  - `loading`
  - `compare`
  - `editor`
  - `manual-fallback`
- [`frontend/src/api/index.ts`](/D:/构想/stock-assistant/frontend/src/api/index.ts) 统一把 `searchStocks()` 解包成数组，消掉页面层的响应结构分叉。

## 6. 验证结果

### 6.1 自动化

- `cd backend && uv run pytest tests/ -v`：`101 passed`
- `cd frontend && pnpm type-check`：通过
- `cd frontend && pnpm build`：通过

### 6.2 交互式 smoke

使用临时本地 backend + Vite，补做了以下验证：

- 设置页保存 `total_capital=500000`，随后 API 读取值为 `500000`
- `/plan` 手动创建计划成功，toast 为“计划已创建”
- `/plan` 编辑 `pending` 计划成功，toast 为“计划已更新”
- `/plan` 将 `pending` 计划标记为“已执行”，编辑入口消失
- `/plan/generate/600519.SH`
  - 通过 Playwright 路由 mock 验证三栏对比
  - 选择方案后进入共享编辑器
  - 修改后保存成功并跳转 `/plan`
- `/plan/generate/000001.SZ`
  - 通过 Playwright 路由 mock 验证 `manual_fallback`
  - 页面展示 warning 并直接进入共享编辑器
- `/plan/generate/600519.SH`
  - 通过 mock `total_capital=0` 验证跳转到设置页并提示“请先在设置页设置总资金”

详细命令和结果见 [`docs/archive/V5/verification.md`](/D:/构想/stock-assistant/docs/archive/V5/verification.md)。

## 8. 剩余风险

- 本轮没有引入正式前端 E2E 测试基础设施，交互路径仍主要由 API 测试、构建检查和 Playwright smoke 共同覆盖。
- 前端构建仍有 Vite chunk size warning，不影响 V5 验收，但后续版本可考虑按路由或大图表模块拆包。
- 本轮严格停留在 V5 范围，不扩到持仓、提醒、复盘等后续版本能力。
