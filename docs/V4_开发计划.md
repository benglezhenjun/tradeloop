# V4 开发计划：LLM 智能分析

编写时间：2026-04-05
基准状态：V3 已完成（仪表盘+市场温度），main 分支为稳定基线

---

## 1. 背景与目标

V3 让用户能看到市场数据。V4 的目标是让系统能**读懂数据、说出结论**：
用户触发分析后，LLM 读取本系统已有的数据，生成一份结构化的文字分析报告。

**V4 核心交付物**：
- 6 个分析 Agent，每个专注一类分析任务
- 一个"分析报告"页面，用户点击触发，等待生成，阅读结果
- 支持两类 LLM Provider：DeepSeek API（云端）和 Ollama（本地），在配置文件中选择

**V4 不做的事**（留到后续版本）：
- 自动定时生成报告（V4 只做手动触发）
- 交易计划和持仓（V5-V6）
- 复杂 Agent 间通信框架（V4 用顺序调用，不引入 LangChain 等重框架）

---

## 2. 6 个 Agent 设计

所有 Agent 的共同结构：
- 输入：从数据库读取的结构化数据（Python dict/list）
- 输出：一段 Markdown 格式的文字分析（str）
- 调用方式：构造 prompt → 调用 LLM → 返回文本

| Agent | 职责 | 数据来源 |
|-------|------|---------|
| `MarketAgent` | 描述当日市场整体环境，判断市场情绪偏多/偏空/震荡 | `get_market_overview()` |
| `IndustryAgent` | 分析行业热度，指出热点板块和弱势板块 | `get_industry_heat()` |
| `WatchlistAgent` | 逐一对自选股近 5 日走势给出简评（仅处理有数据的股票） | `daily_quote` 最近 5 日 |
| `ScreeningAgent` | 解读最近一次策略筛选结果，提炼入选股的共同特征 | 最近一次 `StrategyRun` 及其 `ScreeningResult` |
| `StockAgent` | 针对单只股票，综合近 60 日 K 线走势和财务数据给出深度点评 | `DailyQuote` + `StockFinancial` |
| `ReportAgent` | 汇总前 4 个 Agent 的输出，生成一份完整的每日分析日报 | 以上各 Agent 输出文本 |

**设计约束**：
- `StockAgent` 是单股分析，在个股详情页单独触发，不进入日报流程
- 日报流程：`MarketAgent` → `IndustryAgent` → `WatchlistAgent` → `ScreeningAgent` → `ReportAgent`
- 每个 Agent 独立调用 LLM，失败时记录错误但不中断整体报告（降级为该部分显示错误提示）

## 3. LLM Provider 设计

### 核心设计：供应商无关

所有主流 LLM 服务（DeepSeek、Kimi、OpenAI、本地 Ollama、本地 LM Studio 等）都支持
OpenAI 兼容的 `/v1/chat/completions` 接口。因此代码只需要三个参数：

```
base_url  供应商接口地址
api_key   密钥（本地模型填任意字符串）
model     模型名称
```

**切换供应商 = 改配置文件，代码零改动。**

### 配置（`config/local.toml` 新增）

```toml
[llm]
# 供应商标签，仅用于前端状态展示，不影响调用逻辑
provider = "deepseek"
# 以下三项是实际调用参数，按使用的供应商填写（参考下方预设）
base_url = "https://api.deepseek.com"
api_key  = "sk-xxx"
model    = "deepseek-chat"
```

### 常用供应商预设（复制对应三行替换上方配置即可）

| 供应商 | base_url | model |
|--------|----------|-------|
| DeepSeek（云端） | `https://api.deepseek.com` | `deepseek-chat` |
| Kimi/Moonshot（云端） | `https://api.moonshot.cn/v1` | `moonshot-v1-8k` |
| Ollama（本地） | `http://localhost:11434/v1` | `qwen2.5:14b`（按实际安装的模型名） |
| LM Studio（本地） | `http://localhost:1234/v1` | （LM Studio 界面里显示的模型名） |

**本地模型选型参考**（取决于 GPU 显存）：

| 显存 | 推荐模型 | 说明 |
|------|---------|------|
| 16 GB | `qwen3.5:9b` | 轻松跑满，中文质量好 |
| 16 GB（可接受少量 CPU 溢出） | `gpt-oss:20b` | 13GB，基本能跑 |
| 16 GB（少量 CPU offload） | `qwen3.5:27b` | 17GB 略超，Ollama 会自动溢出到内存 |

本机已安装（`ollama list`）：`qwen3.5:27b`、`qwen3.5:9b`、`gpt-oss:20b`

### 调用方式（统一）

```python
from openai import OpenAI
client = OpenAI(api_key=api_key, base_url=base_url)
response = client.chat.completions.create(model=model, messages=messages)
```

所有供应商完全相同，`base_url` 决定请求去哪里。

### 依赖

后端新增：`openai>=1.0`（通过 `uv add openai` 安装）

## 4. 后端架构

### 新增文件

**`backend/app/services/llm.py`**（LLM 客户端封装）
- `get_llm_client()` → 根据 config 返回配置好的 `OpenAI` 实例
- `chat(messages: list[dict], max_tokens: int) -> str` → 统一调用接口，失败时抛出明确异常

**`backend/app/services/agents.py`**（6 个 Agent 函数）
- `run_market_agent(db) -> str`
- `run_industry_agent(db) -> str`
- `run_watchlist_agent(db) -> str`
- `run_screening_agent(db) -> str`
- `run_stock_agent(db, ts_code: str) -> str`
- `run_report_agent(sections: dict[str, str]) -> str`

每个函数：读数据 → 构造 prompt → 调用 `chat()` → 返回 Markdown 文本

**`backend/app/api/analysis.py`**（分析路由）

```
POST /api/analysis/daily_report
    触发日报生成（MarketAgent + IndustryAgent + WatchlistAgent + ScreeningAgent + ReportAgent）
    同步执行，等待完成后返回完整报告
    返回：{ "report": "...", "sections": { "market": "...", ... }, "generated_at": "..." }

POST /api/analysis/stock/{ts_code}
    触发单股分析（StockAgent）
    返回：{ "ts_code": "...", "analysis": "...", "generated_at": "..." }

GET /api/analysis/llm_status
    检查 LLM 配置是否有效（能否连通）
    返回：{ "configured": true/false, "provider": "deepseek", "model": "deepseek-chat" }
```

### 数据模型

新增 `AnalysisReport` 表（存储历史报告，供前端查看历史）：

```python
class AnalysisReport(Base):
    __tablename__ = "analysis_report"
    id          = Column(Integer, primary_key=True, autoincrement=True)
    report_type = Column(String(20))   # "daily" or "stock"
    ts_code     = Column(String(20), nullable=True)  # 仅 stock 类型有值
    content     = Column(Text)         # Markdown 全文
    sections    = Column(Text)         # JSON 字符串，各 Agent 分段输出
    generated_at = Column(String(30))
```

需要新增 Alembic migration。

## 6. Prompt 设计原则（给 LLM 看）

每个 Agent 的 prompt 结构：
1. **系统角色**：你是一个 A 股市场分析助手，分析专业、简洁、避免废话
2. **数据上下文**：将结构化数据转为自然语言描述（不直接贴 JSON）
3. **输出格式要求**：Markdown 格式，包含标题和要点
4. **限制**：不预测涨跌，不给出买卖建议，只描述现象和可能的成因

示例（MarketAgent）：
```
今日（2026-04-05）市场数据：
- 上涨 2512 家，下跌 1830 家，平盘 156 家
- 涨停 67 家，跌停 8 家
- 全市场成交额 8,432 亿元
- 平均涨跌幅 +0.45%

请分析今日市场整体环境，200字以内，Markdown格式，包含：
1. 市场情绪判断（偏多/偏空/震荡）
2. 成交量情况
3. 值得关注的信号
```

## 7. 工程规范

### 分支管理

```bash
git checkout main
git checkout -b v4-dev
# 初版完成：git tag v4.0-draft
# 打磨完成：git merge v4-dev && git tag v4.0
```

### Alembic

需要新增一个 migration，创建 `analysis_report` 表。

### 超时处理

LLM 调用可能耗时较长（DeepSeek 约 10-30 秒，Ollama 本地约 30-90 秒）。
FastAPI 端点的 timeout 需要足够大，前端 axios timeout 设置为 120 秒。

### 测试策略

- LLM 调用用 mock（测试不真实调用 API）
- 测试覆盖：Agent 函数在有数据/无数据时是否正确构造 prompt 并返回合理结果
- LLM 客户端连通性测试只做手动验证，不写入自动化测试

### 无 LLM 配置时的降级

若 `config/local.toml` 未配置 `[llm]` 或 `api_key` 为空：
- `GET /api/analysis/llm_status` 返回 `{"configured": false}`
- 前端隐藏"生成报告"按钮，显示"请先在设置页配置 LLM"

## 8. 验收标准

1. `GET /api/analysis/llm_status` 返回正确的配置状态
2. 点击"生成今日日报"，等待完成后各 Tab 均有内容（需有效 LLM 配置）
3. 某个 Agent 失败时，其他 Agent 的输出仍然展示，不全页崩溃
4. `StockAgent` 在个股详情页可独立触发和展示
5. 历史报告可查看（生成过的报告保存在数据库，下次打开仍可读）
6. 未配置 LLM 时，前端显示引导提示而非空白或报错
7. `pnpm build` 通过，无 TypeScript 错误
8. `pytest tests/ -v` 全部通过（含新增 mock 测试）

---

## 9. 已知约束与边界

- **不引入 LangChain / LlamaIndex 等重框架**：V4 用直接 API 调用，保持依赖简单
- **报告生成是同步的**：前端 loading 期间用户等待，不做 WebSocket 推送（V4 阶段）
- **Prompt 工程不追求完美**：V4 打基础，prompt 在打磨阶段迭代优化
- **OpenAI 库版本**：使用 `openai>=1.0`，兼容 DeepSeek 和 Ollama 的 `/v1` 接口
