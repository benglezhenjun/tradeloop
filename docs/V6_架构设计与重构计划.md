## 1. 背景与目标

V5 让系统能"做计划"。V6 的目标是让系统能**记录执行、跟踪持仓**：
用户执行了交易计划后，录入买卖记录，系统自动计算盈亏、跟踪持仓，并把计划和实际执行串联起来。

同时，V1-V5 积累的技术债已到必须清理的临界点——本轮在不改变功能的前提下重构底层，为 V7 复盘和 V8+ 量化打好地基。

**V6 = 新功能（交易记录+持仓） + 基础重构（技术债清理）**

**V6 核心交付物**：
- 交易记录 CRUD（买入/卖出/加仓/减仓）
- 记录关联交易计划，执行时自动更新计划状态
- 持仓自动计算（成本价、持仓量、浮盈浮亏）
- 持仓全景页（当前持有 + 已清仓历史）
- 技术债系统性清理（见第 6 节）

**V6 不做的事**（留到后续版本）：
- 复盘与行为模式分析（V7）
- 绩效统计：胜率、盈亏比、月度曲线（V7 复盘里做）
- 自动化预警与通知（V10）
- 回测引擎（V8+）

## 2. 数据模型

### 2.1 TradeRecord 表（新增）

```python
class TradeRecord(Base):
    __tablename__ = "trade_record"
    id              = Column(Integer, primary_key=True, autoincrement=True)
    ts_code         = Column(String(20), nullable=False, index=True)
    stock_name      = Column(String(50), nullable=False)
    plan_id         = Column(Integer, ForeignKey("trading_plan.id"), nullable=True)  # 关联计划（可空=手动录入）
    direction       = Column(String(10), nullable=False)   # "buy" / "sell"
    price           = Column(Float, nullable=False)        # 成交均价
    quantity         = Column(Integer, nullable=False)      # 成交股数（手 × 100）
    amount          = Column(Float, nullable=False)        # 成交金额（price × quantity）
    fee             = Column(Float, nullable=False, default=0)  # 手续费+印花税+过户费
    trade_date      = Column(String(10), nullable=False)   # 成交日期 YYYY-MM-DD
    trade_time      = Column(String(8), nullable=True)     # 成交时间 HH:MM:SS（可空）
    note            = Column(Text, nullable=True)          # 备注
    created_at      = Column(String(30), nullable=False)
```

设计说明：
- `plan_id` 可空：不是所有交易都有对应计划（比如打板、临时操作）
- `quantity` 以**股**为单位（不是手），方便精确计算
- `amount` 冗余存储（= price × quantity），避免前端反复计算
- `fee` 包含所有交易费用，用户手动填（V6 不做券商对接）

### 2.2 Position 表（新增）

```python
class Position(Base):
    __tablename__ = "position"
    id              = Column(Integer, primary_key=True, autoincrement=True)
    ts_code         = Column(String(20), nullable=False, unique=True, index=True)
    stock_name      = Column(String(50), nullable=False)
    total_quantity  = Column(Integer, nullable=False, default=0)    # 当前持仓股数
    avg_cost        = Column(Float, nullable=False, default=0)      # 持仓均价（加权平均）
    total_cost      = Column(Float, nullable=False, default=0)      # 总成本（含手续费）
    realized_pnl    = Column(Float, nullable=False, default=0)      # 已实现盈亏
    status          = Column(String(10), nullable=False, default="holding")  # "holding" / "closed"
    first_buy_date  = Column(String(10), nullable=False)            # 首次建仓日期
    last_trade_date = Column(String(10), nullable=False)            # 最近交易日期
    updated_at      = Column(String(30), nullable=False)
```

设计说明：
- `ts_code` 唯一约束：一只股票只有一条持仓记录（清仓后 status="closed"，再买入时 reopen）
- `avg_cost` 采用**加权平均法**：买入时 `avg_cost = total_cost / total_quantity`
- `realized_pnl` 在卖出时计算：`(sell_price - avg_cost) × sell_quantity - sell_fee`
- `status` 只有两个值：`holding`（持仓中）/ `closed`（已清仓）

### 2.3 计划-记录关联逻辑

```
TradingPlan (1) ←→ (N) TradeRecord
  plan_id 外键关联

录入买入记录 + 关联 plan_id 时：
  → plan.status 自动从 "pending" 变为 "executed"

查看计划详情时：
  → 展示关联的所有交易记录
  → 展示计划 vs 实际的偏差（计划目标价 vs 实际买入价）
```

### 2.4 费用计算公式

```
买入费用 = 成交金额 × 佣金费率（默认万2.5）
卖出费用 = 成交金额 × 佣金费率 + 成交金额 × 印花税率（千1）
过户费   = 成交金额 × 0.00002（上海）

默认费率写入 UserConfig，用户可在设置页修改：
  commission_rate: "0.00025"   # 佣金费率
  stamp_tax_rate:  "0.001"     # 印花税率（仅卖出）
  transfer_fee_rate: "0.00002" # 过户费率
```

### 3.2 核心业务逻辑：录入交易记录

```python
def create_trade(db, data):
    # 1. 校验输入（复用 contract 模式）
    # 2. 计算 amount = price × quantity
    # 3. 自动计算费用（如用户未填，按默认费率算）
    # 4. 保存 TradeRecord
    # 5. 更新 Position：
    #    - 买入：total_quantity += quantity, total_cost += amount + fee
    #            avg_cost = total_cost / total_quantity
    #    - 卖出：realized_pnl += (price - avg_cost) × quantity - fee
    #            total_quantity -= quantity
    #            if total_quantity == 0: status = "closed"
    # 6. 如有 plan_id 且 direction == "buy"：
    #    plan.status = "executed"（如果当前是 pending）
    # 7. 返回记录 + 更新后的持仓快照
```

## 5. 交互流程

### 5.1 从计划录入交易（主流程）

```
用户在 PlanView 看到一个 pending 计划
  → 点击"录入交易"
  → 跳转 TradeForm，自动预填：股票、方向=buy、目标价（作为参考）、关联计划ID
  → 用户填写实际成交价、数量、日期
  → 费用自动按费率计算，显示在表单里，用户可修改
  → 提交 → POST /api/trade
    → 后端保存记录 + 更新持仓 + 计划状态变为 executed
  → 前端跳转到持仓页，看到新增持仓
```

### 5.2 手动录入交易（无计划）

```
用户在 TradeRecordView 点击"录入交易"
  → 打开 TradeForm（空表单）
  → 搜索选股、填方向、价格、数量、日期
  → 关联计划留空
  → 提交
```

### 5.3 卖出 / 减仓

```
用户在 PositionView 持仓卡片上点击"卖出"
  → 跳转 TradeForm，预填股票、方向=sell、当前持仓量作为参考
  → 用户填实际卖出价、数量
  → 提交 → 后端扣减持仓、计算已实现盈亏
  → 如果 total_quantity == 0 → 持仓状态变为 closed
```

### 5.4 删除交易记录

```
删除记录 → 回滚持仓变动（重新计算该股所有剩余记录）
  → 使用 recalculate_position(ts_code) 从零重算
  → 避免逆向计算的精度和逻辑问题
```

## 6. 技术债清理（随 V6 一并执行）

以下重构**不改变任何功能**，只改善代码质量和可维护性。

### 6.1 后端重构

#### R1：提取单位转换常量

**现状**：`/ 100_000`（千元→亿元）和 `/ 10_000`（万元→亿元）散落在 10+ 个文件共 20+ 处。
**目标**：新建 `backend/app/constants.py`，定义 `AMOUNT_UNIT_TO_YI = 100_000` 和 `MV_UNIT_TO_YI = 10_000`，全局替换。
**涉及文件**：`screening.py`, `stocks.py`, `kline.py`, `market.py`, `agents.py`, `watchlist.py`, `conditions/amount_gt.py`, `conditions/market_cap_gt.py`

#### R2：统一错误响应契约

**现状**：`plan.py` 用 `{"error", "error_type"}`，其他服务用 `{"error"}` 裸 dict，API 层各自处理。
**目标**：将 `plan_contract.py` 中的 `build_error()` 提升为全局共享 `app/errors.py`，所有服务统一使用。API 层统一 `_raise_service_error()` 辅助函数。
**涉及文件**：所有 `api/*.py` + `services/*.py`

#### R3：消除原始 SQL 元组访问

**现状**：`stocks.py`, `watchlist.py` 等文件使用 `db.execute(text(...)).fetchone()` 返回元组，用 `row[0]`, `row[1]` 取值。
**目标**：改用 `.mappings().fetchone()` 返回字典，或直接使用 ORM `db.query()` / `db.get()`。
**涉及文件**：`api/stocks.py`, `services/watchlist.py`, `services/market.py`

#### R4：agents.py 拆分与 MA 计算复用

**现状**：`agents.py` 有 600+ 行，6 个 Agent + 1 个 PlanAgent 全在一个文件。StockAgent 和 PlanAgent 各自计算 MA5/MA20/MA60。
**目标**：
1. 提取 `services/utils/indicators.py`，包含 `calc_ma(closes, period)` 和 `calc_price_stats(quotes)` 工具函数
2. 将 `agents.py` 拆为 `agents/base.py`（共享工具）+ `agents/daily.py`（6 个日报 Agent）+ `agents/plan.py`（PlanAgent）
**结果**：每个文件 200 行左右，MA 逻辑只写一次

#### R5：Pydantic 模型加强输入校验

**现状**：`api/plan.py` 的 `StatusUpdate` 接受任意字符串，靠服务层校验。
**目标**：所有 API 输入模型使用 `Literal` 类型约束枚举值。
```python
class StatusUpdate(BaseModel):
    status: Literal["executed", "abandoned"]
```
**涉及文件**：`api/plan.py`, 新增的 `api/trade.py`, `api/position.py`

### 6.2 前端重构

#### R6：API baseURL 环境变量化

**现状**：`api/index.ts` 硬编码 `http://localhost:8000`。
**目标**：改读 `import.meta.env.VITE_API_URL`，`.env.development` 写默认值。
**涉及文件**：`api/index.ts`, 新增 `.env.development`

#### R7：Plan Store 拆分

**现状**：`stores/plan.ts` 同时管理计划列表、生成上下文、编辑草稿三类状态。
**目标**：保留 `stores/plan.ts`（计划 CRUD），提取 `stores/planGenerate.ts`（生成流程状态）。PlanView 和 PlanGenerateView 各用各的 store。
**涉及文件**：`stores/plan.ts`, `views/PlanView.vue`, `views/PlanGenerateView.vue`

## 9. 已知约束与边界

- **不与券商对接**：所有交易数据手动录入，V6 不做自动导入
- **费用是估算**：按默认费率计算，用户可手动覆盖，不保证精确到分
- **浮盈浮亏依赖行情数据**：如果当日未同步数据，用最近一个交易日的收盘价
- **不做盈亏归因**：只计算简单的买卖差价盈亏，不分解为 alpha/beta（留 V8+）
- **持仓不分账户**：V6 只有一个默认账户，不支持多账户区分
