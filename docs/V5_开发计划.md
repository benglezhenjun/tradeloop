# V5 开发计划：交易计划 + LLM 风控

编写时间：2026-04-06
基准状态：V4 已合入 main（v4.0 标签），v5-dev 分支开发

---

## 1. 背景与目标

V4 让系统能"读懂数据、说出结论"。V5 的目标是让系统能**辅助决策**：
用户看完分析报告后，针对感兴趣的股票，LLM 生成三套交易方案（激进/稳健/保守），
用户选择、编辑、保存为自己的交易计划，并跟踪执行状态。

**V5 核心交付物**：
- LLM 生成三套交易方案（激进/稳健/保守），含仓位计算、分批止盈、风控评语
- 交易计划 CRUD（创建、查看、编辑、状态管理）
- 手动创建计划（不依赖 LLM）
- 全局总资金设置
- 交易计划管理页（PlanView）+ 个股详情页快捷入口

**V5 不做的事**（留到后续版本）：
- 交易记录与持仓跟踪（V6）
- 复盘与行为模式分析（V7）
- 计划到期自动提醒（V10 通知中心）

## 2. 数据模型

### 2.1 TradingPlan 表

```python
class TradingPlan(Base):
    __tablename__ = "trading_plan"
    id              = Column(Integer, primary_key=True, autoincrement=True)
    ts_code         = Column(String(20), nullable=False)   # 关联股票代码
    stock_name      = Column(String(50), nullable=False)   # 股票名称（冗余存储，避免频繁联查）
    direction       = Column(String(10), nullable=False)   # "buy" / "sell"
    target_price    = Column(Float, nullable=False)        # 目标入场价
    stop_loss_price = Column(Float, nullable=False)        # 止损价
    take_profit     = Column(Text, nullable=False)         # JSON: 分批止盈档位
    position_ratio  = Column(Float, nullable=False)        # 仓位比例（0-0.4），占总资金百分比
    reasoning       = Column(Text, nullable=False)         # 计划理由/逻辑（Markdown）
    risk_comment    = Column(Text, nullable=True)          # LLM 风控评语
    tier_label      = Column(String(20), nullable=True)    # "aggressive" / "balanced" / "conservative"
    source          = Column(String(20), nullable=False)   # "llm_generated" / "manual"
    status          = Column(String(20), nullable=False, default="pending")  # "pending" / "executed" / "abandoned"
    expiry_date     = Column(String(10), nullable=True)    # 有效期，格式 YYYY-MM-DD
    alternatives    = Column(Text, nullable=True)          # JSON: 未被选中的 LLM 备选方案
    created_at      = Column(String(30), nullable=False)
    updated_at      = Column(String(30), nullable=False)
```

### 2.2 take_profit JSON 结构（分批止盈）

LLM 建议档数和参数，用户可编辑：

```json
[
  { "price": 25.50, "ratio": 0.3, "note": "第一档：突破前高减仓30%" },
  { "price": 27.00, "ratio": 0.4, "note": "第二档：目标位减仓40%" },
  { "price": 29.00, "ratio": 0.3, "note": "第三档：剩余全部清仓" }
]
```

- `price`：止盈触发价
- `ratio`：该档卖出占该计划持仓的比例（所有档位 ratio 之和 = 1.0）
- `note`：LLM 给出的说明

### 2.3 alternatives JSON 结构（备选方案）

存储用户未选中的两套方案，供复盘参考：

```json
[
  {
    "tier_label": "aggressive",
    "target_price": 24.00,
    "stop_loss_price": 22.50,
    "take_profit": [...],
    "position_ratio": 0.35,
    "reasoning": "...",
    "risk_comment": "..."
  },
  {
    "tier_label": "conservative",
    ...
  }
]
```

### 2.4 UserConfig 表（全局配置）

```python
class UserConfig(Base):
    __tablename__ = "user_config"
    key   = Column(String(50), primary_key=True)   # 配置键
    value = Column(Text, nullable=False)            # 配置值（字符串存储）
```

初始配置项：
- `total_capital`：总资金（元），默认 `"0"`，用户在设置页手动填写

## 3. LLM 方案生成 Agent

### 3.1 PlanAgent（新增）

**职责**：针对单只股票，结合市场环境和个股数据，生成三套交易方案。

**输入数据**：
- 个股近 60 日 K 线（OHLCV + MA）
- 个股财务数据（PE、PB、ROE 等）
- 当日市场概览（涨跌家数、成交额）
- 用户总资金

**输出**：三套方案（激进/稳健/保守），每套包含：
- 方向（买入/卖出）
- 目标入场价 + 入场理由
- 止损价 + 止损逻辑
- 分批止盈档位（LLM 自行决定分几档）
- 建议仓位比例（不超过 40%）+ 仓位理由
- 风控评语（整体风险评估）

**Prompt 结构**：

```
你是一个 A 股交易计划助手。根据以下数据，为该股票生成三套交易方案。

## 股票信息
- 代码：{ts_code}，名称：{name}，行业：{industry}
- 最新价：{close}，PE：{pe}，PB：{pb}，ROE：{roe}

## 近60日K线摘要
{k_line_summary}

## 当日市场环境
{market_summary}

## 用户资金
- 总资金：{total_capital} 元

## 输出要求
生成三套方案，分别标记为"激进"、"稳健"、"保守"。
每套方案必须包含：
1. 方向（买入/卖出）
2. 目标入场价及理由
3. 止损价及逻辑（明确百分比）
4. 分批止盈计划（自行决定档数，每档注明价格、卖出比例、触发条件）
5. 建议仓位比例（占总资金，上限40%）及理由
6. 风控评语（50字以内的风险提示）

三套方案的风险收益比应有明显梯度。
输出格式：严格按 JSON 格式，便于程序解析。
```

### 3.2 输出解析

LLM 返回 JSON，后端解析为三个方案对象。若 JSON 解析失败，降级为纯文本存储并提示用户手动创建。

## 4. 后端架构

### 4.1 新增文件

**`backend/app/models/plan.py`**
- `TradingPlan` 模型
- `UserConfig` 模型

**`backend/app/services/plan.py`**（交易计划服务）

- `create_plan(db, plan_data) -> TradingPlan` — 创建计划（LLM 选定或手动）
- `update_plan(db, plan_id, updates) -> TradingPlan` — 编辑计划

- `get_plans(db, filters) -> list[TradingPlan]` — 计划列表（支持按状态/股票筛选）
- `get_plan_detail(db, plan_id) -> TradingPlan` — 计划详情

**`backend/app/services/user_config.py`**（用户配置服务）
- `get_config(db, key) -> str | None`
- `set_config(db, key, value)`

**`backend/app/api/plan.py`**（交易计划路由）

```
POST   /api/plan/generate/{ts_code}     — LLM 生成三套方案（返回三套，不落库）
POST   /api/plan                         — 保存计划（用户选定/编辑后提交）
GET    /api/plan                         — 计划列表（query: status, ts_code）
GET    /api/plan/{plan_id}               — 计划详情
PUT    /api/plan/{plan_id}               — 编辑计划
PATCH  /api/plan/{plan_id}/status        — 变更状态（pending→executed / pending→abandoned）
DELETE /api/plan/{plan_id}               — 删除计划
```

**`backend/app/api/user_config.py`**（用户配置路由）

```
GET    /api/config/{key}                 — 获取配置
PUT    /api/config/{key}                 — 设置配置
```

## 6. 交互流程

### 6.1 LLM 生成流程

```
用户在个股详情页点击"生成交易计划"
  → 跳转 /plan/generate/{ts_code}
  → 前端检查总资金是否已设置（未设置则提示去设置页填写）
  → POST /api/plan/generate/{ts_code}
    → PlanAgent: 读取 K线 + 财务 + 市场数据 + 总资金
    → LLM 生成三套 JSON 方案
    → 返回三套方案（不落库）
  → 前端三栏展示
  → 用户选择一套 → 编辑调整
  → POST /api/plan（含 alternatives 字段存另外两套）
  → 落库，状态 pending
  → 跳转到计划列表页
```

### 6.2 手动创建流程

```
用户在计划列表页点击"手动创建"
  → 弹窗/表单：选股票、填方向、目标价、止损价、止盈档位、仓位比例、理由
  → POST /api/plan（source = "manual"，无 alternatives）
  → 落库
```

### 6.3 状态变更流程

```
用户在计划卡片上点击"标记已执行" / "放弃"
  → PATCH /api/plan/{id}/status { status: "executed" / "abandoned" }
  → 更新状态 + updated_at
```

## 8. 验收标准

1. 设置页可填写总资金，保存后持久化
2. 个股详情页点击"生成交易计划"，LLM 返回三套方案，三栏对比展示
3. 用户选择方案后可编辑所有字段（价格、止盈档位增删改、仓位滑块）
4. 保存后跳转计划列表页，新计划状态为"待执行"
5. 备选方案在计划详情中可折叠查看
6. 手动创建计划功能正常（不调用 LLM）
7. 状态流转正确（待执行 → 已执行 / 已放弃），不可逆向
8. 仓位比例不超过 40%（前端滑块 + 后端校验双重限制）
9. LLM 返回 JSON 解析失败时，给出友好提示而非崩溃
10. 未设置总资金时，生成按钮提示"请先设置总资金"
11. `pnpm build` 通过，无 TypeScript 错误
12. `pytest tests/ -v` 全部通过（含新增 test_v5_plan.py）

## 9. 已知约束与边界

- **LLM 输出 JSON 不保证格式正确**：必须做 try/except 解析，失败时降级处理
- **仓位比例是建议值**：系统不做实际资金划转，仅记录计划
- **总资金是用户手动填的**：不与券商系统对接，不自动同步
- **计划不自动过期**：到期日仅供用户参考，不自动变更状态（自动提醒留到 V10）
- **不做回测验证**：V5 只管"计划"，不验证计划是否盈利（V7 复盘做）
